# 前端接口联调说明

本文档面向 H5 会员端、配送端、管理端前端，便于与后端联调与测试。

## 1. 基础信息

| 项 | 说明 |
|----|------|
| 默认服务地址 | 按部署为准，例如 `http://服务器IP:8001` 或本地 `http://127.0.0.1:8001` |
| API 前缀 | 业务接口统一为 **`/api`** |
| 协议与格式 | **HTTPS/HTTP** + **`Content-Type: application/json`**（`GET` 无 Body） |
| 健康检查 | `GET /health`，成功体见 §1.4，**无需** `/api` 前缀 |
| 在线文档 | 服务启动后可访问 **`/docs`**（Swagger UI）、**`/redoc`**，与实现保持一致 |

**文档说明**：下文每个接口均配有「接口说明」，概括用途与调用注意；字段细节以 `/docs` 中的模型为准。

### 1.1 跨域（CORS）

浏览器请求需在服务端 `.env` 中配置 **`CORS_ORIGINS`**（逗号分隔的前端 Origin，例如 `https://h5.example.com`）。需带 Cookie 时前端请求设置 `credentials: 'include'`；当前会员/配送/管理鉴权主要使用 **Authorization头**，一般可不依赖 Cookie。

### 1.2 鉴权（JWT）

除特别说明外，需在请求头携带：

```http
Authorization: Bearer <access_token>
```

登录类接口成功时，令牌在 **统一响应的 `data` 字段** 内（见 §1.4），例如：

```json
{
  "code": 200,
  "data": {
    "access_token": "eyJ...",
    "token_type": "bearer"
  },
  "msg": "登录成功"
}
```

前端保存 `data.access_token`，后续请求放入 `Authorization`。Token 过期后需重新登录（会员走微信小程序登录、配送员/管理员走各自登录接口）。

**角色说明**（载荷内含 `role`，前端通常只需按端使用对应 Token，勿混用）：

- 会员：`role` 为 `member`，`sub` 为 **`members.id`（数字主键）**
- 配送员：`role` 为 `courier`，`sub` 为工号 `courier_id`
- 管理员：`role` 为 `admin`，`sub` 为用户名

### 1.3 错误响应

HTTP 4xx/5xx 时，响应体统一为 **`{ "code", "data", "msg" }`**：

- **`code`**：失败时与 HTTP 状态码一致（如 400、401、422、429、502、500）。成功时见 §1.4，**`code` 固定为 200**（与 HTTP 一致）。
- **`data`**：失败时一般为 **`null`**。
- **`msg`**：人类可读说明；参数校验失败时可能为拼接后的字段错误摘要。

示例：

```json
{
  "code": 400,
  "data": null,
  "msg": "请求参数无效"
}
```

**429**：触发了接口限流（见 §1.5）。**502** 在微信等上游服务异常时可能出现。

### 1.4 成功响应（统一信封）

除文档特别说明外，成功时 HTTP **200**，body 形如：

```json
{
  "code": 200,
  "data": {},
  "msg": "获取成功"
}
```

- 无额外载荷的接口（如仅提示成功）可为 `"data": null`。
- 下文各接口的「响应示例」如只写业务对象，均指 **`data` 内的内容**；联调时请按完整信封解析。

### 1.5 接口限流（防刷）

以下接口按 **客户端 IP** 限流（单机内存计数，多实例部署时需在网关层叠加限流）：

| 接口 | 默认限制 |
|------|----------|
| `POST /api/user/wx/mini/login` | 30 次/分钟 |
| `POST /api/user/register` | 20 次/分钟 |
| `POST /api/admin/login` | 30 次/分钟 |
| `POST /api/courier/login` | 30 次/分钟 |

超限返回 **HTTP 429**，body 同 §1.3（`code`: 429）。

## 2. 会员端（H5）

**章节说明**：以下接口供会员 H5 使用；除标明「无需登录」外，均须在请求头携带会员 JWT（§1.2）。

### 2.1 微信小程序登录

`POST /api/user/wx/mini/login`  
**无需登录**

**接口说明**：`js_code` 来自 `wx.login()`；`phone_code` 来自手机号授权组件 `getPhoneNumber` 回调。服务端校验微信凭证、换取手机号，按手机号匹配或创建会员，并写入/更新 `members.wx_mini_openid`，签发会员 JWT（§1.2）。需在服务端配置 `WX_MINI_APPID`、`WX_MINI_SECRET`。

请求体：

```json
{
  "js_code": "081xxxxx",
  "phone_code": "e31968a7f94cc5ee25fafc2aef2773f0xxxx"
}
```

响应：见 §1.2，`data` 内为 `access_token` / `token_type`。未配置小程序或微信接口失败时可能为 **503** / **502**。

---

### 2.2 新成员登记

`POST /api/user/register`  
**无需登录**

**接口说明**：新会员首次提交姓名、地址等资料完成建档。`remarks` 为可选。**配送区域 `area` 由服务端根据地址高德地理编码后的坐标（GCJ-02），与后台配置的配送区域多边形自动匹配**；未匹配或无法地理编码时为 `"未分配"`。若手机号已注册，以接口实际错误为准。

请求体：

```json
{
  "phone": "13800138000",
  "name": "张三",
  "address": "浙江省杭州市西湖区xx路xx号",
  "remarks": "不吃葱"
}
```

`remarks` 可选。无需再传 `area`。

响应：`code === 200`，`data` 为会员详情对象，结构同 §2.4（`MemberOut`），其中 `area` 为字符串（区域名或 `未分配`）。

---

### 2.3 激活计划

`POST /api/user/activate`  
**需要会员 Token**（身份以 JWT `sub` = `members.id` 为准）。

**接口说明**：将会员套餐/计划置为「已激活」状态。Body 可为空对象 `{}`。

请求体：

```json
{}
```

响应：`data` 为会员详情对象。

---

### 2.4 查询当前会员信息

`GET /api/user/me`  
**需要会员 Token**。

**接口说明**：返回当前 Token 对应会员完整档案（余额、套餐类型、请假区间、默认地址与坐标等）。

响应示例（`data` 内为会员对象；字段名固定，日期类在 `leave_range` 内）：

```json
{
  "code": 200,
  "msg": "获取成功",
  "data": {
    "phone": "13800138000",
    "name": "张三",
    "address": "……",
    "location": { "lng": 120.12345678, "lat": 30.12345678 },
    "area": "东区",
    "remarks": "不吃葱",
    "balance": 10,
    "plan_type": "次卡",
    "is_active": true,
    "is_leaved_tomorrow": false,
    "tomorrow_leave_target_date": null,
    "leave_range": { "start": "2026-04-01", "end": "2026-04-05" },
    "leave_deadline_time": "21:00:00",
    "created_at": "2026-04-14T12:00:00"
  }
}
```

说明：`location` 可能为 `null`（未地理编码成功）；`plan_type`、`leave_range`、`tomorrow_leave_target_date` 可能为 `null`；`leave_deadline_time` 为后台配置的当日请假截止时刻（`HH:MM:SS`），供小程序与后端校验一致。

---

### 2.5 请假 / 取消请假

`POST /api/user/leave`  
**需要会员 Token**（身份以 JWT `sub` = `members.id` 为准）。

**接口说明**：提交「仅明天请假」「按日期区间请假」或「取消请假」。`tomorrow` 与 `range` 均受管理端配置的「请假截止时间」约束（上海业务日当日时刻前）；`range` 须同时传 `start`、`end`。

请求体：

```json
{
  "type": "tomorrow",
  "start": null,
  "end": null
}
```

**`type` 取值**：

| 值 | 含义 |其它字段 |
|----|------|----------|
| `tomorrow` | 仅明天请假 | 无需 `start`/`end`；受后台「请假截止时间」约束 |
| `range` | 区间请假 | **必填** `start`、`end`，格式 `YYYY-MM-DD`；受「请假截止时间」约束；提交后会清除「明天请假」标记 |
| `clear_tomorrow` | 仅取消明天请假 | 不影响已设置的区间请假 |
| `cancel` | 取消全部请假 | 清空「明天请假」与「区间请假」 |

响应：`data` 为会员详情对象。

---

### 2.6 菜单（可未登录访问）

菜品在**商品库**（`menu_dish`）维护，并可归属**分类**（如 `code=weekly` 的「每周餐品」）。排期有两种：

- **周槽位** `weekly_menu_slot`：存 `week_start`（当周周一）+ `slot`（1=周一 … 7=周日）+ `dish_id`。**「本周 / 下周」不单独建表**：查询时用「今天所在周的周一」与「+7 天的周一」作为两个 `week_start` 即可；日历翻周后，原「下周」那组行自然变成「本周」，**无需定时任务拷贝**。
- **按日排期** `menu_schedule`：单日绑定（兼容/补缺）；会员接口**优先周槽位**，缺日再回退按日表。

同一自然月内同一 `dish_id` 仅可出现一次（周槽位与按日表均受约束，具体以后端为准）。

#### 2.6.1 明日餐谱

`GET /api/menu/tomorrow`

**接口说明**：获取「明天」（上海业务日）排期对应的菜品；无排期时各字段多为 `null`。

响应示例：

```json
{
  "code": 200,
  "msg": "获取成功",
  "data": {
    "date": "2026-04-15",
    "dish_id": 12,
    "title": "红烧肉套餐",
    "desc": "……",
    "pic": "https://..."
  }
}
```

#### 2.6.2 一周菜单

`GET /api/menu/weekly`  
可选查询参数：`week_start`（`YYYY-MM-DD`，该周任意一天，后端归一为当周周一；不传则为上海「今天」所在周）。

`data.week_start` 为周一日期字符串；`data.items` 长度恒为 7。每项含 `date`、`slot`（1–7）、`dish_id`（无排期为 `null`）、`title`、`desc`、`pic`。

#### 2.6.3 餐品详情

`GET /api/menu/detail/{dish_id}`

`dish_id` 为菜品库主键（与周列表项中 `dish_id` 一致）。`data` 含 `dish_id`、`title`、`desc`、`pic`、`is_enabled`、`category_id`（可 `null`）。

---

### 2.7 地图跳转（前端拼接，非后端接口）

**说明**：非服务端接口。由前端根据会员地址或坐标拼接地图 URI，唤起导航（坐标须为 GCJ-02）。

高德 Web 示例：`https://uri.amap.com/marker?position=<lng>,<lat>&name=<URL编码后的地址>`

## 3. 配送端

**章节说明**：以下接口供配送员 App/H5 使用；除登录外均需携带配送员 JWT。

### 3.1 配送员登录

`POST /api/courier/login`  
**无需登录**

**接口说明**：配送员使用工号 `courier_id` 与 PIN 登录，换取 JWT；后续 `tasks`、`confirm` 均使用该 Token。

请求体：

```json
{
  "courier_id": "C001",
  "pin": "123456"
}
```

响应：见 §1.2，`data` 内为令牌字段。

---

### 3.2 当日配送清单

`GET /api/courier/tasks?area=东区`  
**需要配送员 Token**

**接口说明**：查询当前业务日（通常为当天）的配送任务列表。可选 `area` 只拉取某一片区；不传则返回按区域分组的全量数据，便于分片区浏览。

查询参数：

- `area`（可选）：`东区` | `南边` | `北边` | `西边`；不传则返回按区域分组后的全部数据。

响应示例（业务数据在 `data` 内）：

```json
{
  "code": 200,
  "msg": "获取成功",
  "data": {
    "delivery_date": "2026-04-14",
    "groups": [
      {
        "area": "东区",
        "items": [
          {
            "phone": "13800138000",
            "name": "张三",
            "address": "……",
            "lng": 120.12,
            "lat": 30.12,
            "area": "东区",
            "remarks": "放前台",
            "sort_distance_m": 1250.5
          }
        ]
      }
    ]
  }
}
```

若指定了 `area`，`groups` 通常仅一组。`sort_distance_m` 可能为 `null`（无坐标时）。

---

### 3.3 确认送达

`POST /api/courier/confirm`  
**需要配送员 Token**

**接口说明**：将指定会员在指定配送日标记为「已送达」，用于扣次与闭环。`date` 与 `delivery_date` 二选一、含义相同；不传则默认**上海业务日**当天。同一用户同一配送日重复确认**幂等**，不会重复扣次。

请求体（`date` 与 `delivery_date` 等价，任选其一；不传表示**上海业务日**当天）：

```json
{
  "user_phone": "13800138000",
  "date": "2026-04-14"
}
```

或：

```json
{
  "user_phone": "13800138000",
  "delivery_date": "2026-04-14"
}
```

当日送达可简写：

```json
{ "user_phone": "13800138000" }
```

成功响应示例：

```json
{
  "code": 200,
  "data": null,
  "msg": "送达已确认"
}
```

说明：同一用户同一配送日重复确认为幂等，不会重复扣次。

## 4. 管理端

**章节说明**：以下接口供后台管理使用；除登录外均需管理员 JWT。

### 4.1 管理员登录

`POST /api/admin/login`  
**无需登录**

**接口说明**：管理员用户名 + 密码登录，换取 JWT；后续用户列表、充值、菜单、设置等均使用该 Token。

请求体：

```json
{
  "username": "admin",
  "password": "你的密码"
}
```

响应：见 §1.2，`data` 内为令牌字段。

---

### 4.2 用户列表（分页 + 搜索）

`GET /api/admin/users?q=138&page=1&page_size=20`  
**需要管理员 Token**

**接口说明**：分页查询会员列表，支持按手机号关键字 `q` 模糊筛选，用于后台用户管理与检索。

查询参数：

- `q`（可选）：按手机号模糊匹配
- `page`：页码，默认 `1`
- `page_size`：每页条数，默认 `20`

响应示例（列表与分页字段均在 `data` 内）：

```json
{
  "code": 200,
  "msg": "获取成功",
  "data": {
    "items": [
      {
        "phone": "13800138000",
        "name": "张三",
        "address": "……",
        "area": "东区",
        "balance": 10,
        "plan_type": "次卡",
        "is_active": true,
        "created_at": "2026-04-14T12:00:00"
      }
    ],
    "total": 100,
    "page": 1,
    "page_size": 20
  }
}
```

---

### 4.3 充值 / 调整次数

`POST /api/admin/recharge`  
**需要管理员 Token**

**接口说明**：对指定会员增加或减少剩余次数（`amount` 正负），并可同步调整 `plan_type`（次卡/周卡/月卡）。常用于开户、续费或人工纠错。

请求体：

```json
{
  "phone": "13800138000",
  "amount": 10,
  "plan_type": "月卡"
}
```

- `amount`：正数为增加次数，负数为扣减
- `plan_type`（可选）：`"次卡"` | `"周卡"` | `"月卡"`

响应：`data` 与会员详情结构一致（§2.4），便于刷新界面。

---

### 4.4 菜品库、分类与排期

**需要管理员 Token**。

#### 4.4.1 分类列表

`GET /api/admin/categories`  
含内置 `weekly`（每周餐品）。新建分类：`POST /api/admin/category`， body：`code`、`name`、`sort_order`。

#### 4.4.2 菜品列表

`GET /api/admin/dishes`  
可选查询参数：`enabled_only=true` 只返回启用菜品。

#### 4.4.3 新建 / 更新菜品

`POST /api/admin/dish`

新建时省略 `id`；更新时带 `id`。可选 `category_id` 归属分类。请求体示例：

```json
{
  "id": 12,
  "name": "红烧肉套餐",
  "description": "配料说明",
  "image_url": "https://example.com/1.jpg",
  "is_enabled": true,
  "category_id": 1
}
```

响应 `data` 为菜品对象（含 `id`、`category_id`、`created_at` 等）。

删除菜品：`DELETE /api/admin/dish/{dish_id}`。若仍被周槽位或按日排期引用，返回 `409`。

列表支持按名称筛选：`GET /api/admin/dishes?q=关键词`。

#### 4.4.4 每周槽位（本周 / 下周预告）

- `GET /api/admin/menu/weekly-slots`：不传 `week_start` 时，返回 `this_week_start`、`next_week_start` 及两组槽位列表 `this_week`、`next_week`（每项含 `slot`、`dish_id`、`name` 等）；传 `week_start`（任意一天）则只返回该周。
- `POST /api/admin/menu/weekly-slot`：body：`week_start`（任意一天，服务端归一为周一）、`slot`（1–7）、`dish_id`（可选；省略或 `null` 表示清空该槽）。同月同一菜多占会 `409`。

#### 4.4.5 按日排期（补缺 / 兼容）

`POST /api/admin/menu/schedule`

将某日绑定到菜品库中的 `dish_id`。**同一自然月内，同一道菜只能排一天**；违反时返回 `409`。

```json
{
  "date": "2026-04-15",
  "dish_id": 12
}
```

---

### 4.5 全局设置（请假截止时间）

`POST /api/admin/settings`  
**需要管理员 Token**

**接口说明**：配置会员提交「明天请假」与「区间请假」的截止时刻（上海当日 `time`）；超过后会员无法再提交 `type=tomorrow` 或 `type=range`（管理端代操作不校验；具体以后端为准）。

请求体（时间字符串，建议 `HH:MM:SS`）：

```json
{ "leave_deadline_time": "21:00:00" }
```

成功：`code === 200`，`msg` 提示更新成功（`data` 可为 `null`）。

---

### 4.6 修改会员地址（并触发地理编码）

`POST /api/admin/member/address`  
**需要管理员 Token**

**接口说明**：后台修正会员配送地址全文，并触发地理编码以更新经纬度；配送端 `tasks` 中的坐标与排序距离依赖该数据。

请求体：

```json
{
  "phone": "13800138000",
  "address": "新地址全文"
}
```

响应：`data` 为会员详情对象（§2.4）。**成功更新地址后，服务端会按新坐标重新匹配配送区域并写回 `area`（未匹配则为 `未分配`）。**

---

### 4.7 配送区域（CRUD + 绑定配送员）

均需管理员 Token。多边形 `polygon_json` 为 GCJ-02，可为坐标数组 `[[lng,lat],...]`（至少 3 点）或 GeoJSON `Polygon` / `Feature`。

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/api/admin/delivery-regions` | 列表（含 `couriers` 绑定） |
| GET | `/api/admin/delivery-regions/{id}` | 详情 |
| POST | `/api/admin/delivery-regions` | 创建 |
| PATCH | `/api/admin/delivery-regions/{id}` | 部分更新；若传 `couriers` 则**全量替换**该区域绑定 |
| DELETE | `/api/admin/delivery-regions/{id}` | 删除区域（级联删除绑定） |

创建/更新请求体示例（POST全部字段；PATCH 均可选）：

```json
{
  "name": "东片区",
  "code": "E1",
  "polygon_json": [[120.1, 30.1], [120.2, 30.1], [120.15, 30.2]],
  "priority": 0,
  "is_active": true,
  "couriers": [
    { "courier_id": "C001", "is_primary": true, "sort_order": 0 },
    { "courier_id": "C002", "is_primary": false, "sort_order": 1 }
  ]
}
```

每个区域**至多一名** `is_primary: true` 的配送员。

### 4.8 配送员列表（管理端选用）

`GET /api/admin/couriers`**需要管理员 Token**

**接口说明**：返回全部配送员工号与姓名，供配置配送区域时多选。

响应 `data` 为数组：`{ "courier_id", "name", "is_active" }`。

## 5. 联调顺序建议

1. `GET /health` → 确认服务已启动（`code === 200`，`data.status === "ok"`）。  
2. **管理端**：登录 → 发布明日菜单 → 必要时 `settings`、给用户 `recharge`。  
3. **会员端**：`wx/mini/login` → `GET /api/user/me`；新用户可先 `register` 再登录 → `activate` / `leave`。  
4. **配送端**：`login` → `tasks` → `confirm`。  
5. 浏览器联调前确认 **CORS** 与 **HTTPS/混合内容** 策略。
